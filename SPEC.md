# 技術ブログ週次ダイジェストアプリ MVP 仕様

## 1. 目的

自分が普段巡回している技術ブログを対象に、記事を自動収集し、要約・分類・スコアリングを行い、**カレンダー週ごとにまとまった週次ダイジェスト** として閲覧できる Web アプリを作る。

この MVP の主目的は、最終的に欲しい「週報」の前段として、**週単位で記事を整理・要約した体験が本当に有効かを検証すること** にある。

---

## 2. MVP のゴール

- 自分が登録した技術ブログから記事を自動収集できる
- 記事ごとに要約・トピック付与・スコアリングが行われる
- 記事はカレンダー週単位で整理される
- 週ごとに短いサマリーと注目トピックが表示される
- 記事一覧は総合スコア順で並び、理由も表示される
- この体験が将来的な「週報」につながるだけの価値を持つか確認できる

---

## 3. MVP でやること / やらないこと

### やること
- 技術ブログの自動収集
- 記事本文の抽出
- 記事ごとの要約生成
- 記事ごとのトピック付与
- 記事ごとのスコアリング
- カレンダー週単位の整理
- 週サマリー生成
- 注目トピック生成
- 週次ダイジェスト画面の表示

### やらないこと
- X のポスト収集
- GitHub PR の収集
- 複数ユーザー対応
- 高度な推薦学習
- 行動ログによる自動最適化
- 永続化された正式な「週報」生成
- 高度なエラーハンドリング設計
- 類似記事の意味的統合
- Web 全体からの探索型収集

- 自分専用
- 普段巡回している技術ブログを効率よく追いたい人
- 記事を全文読む前に「今週読む価値が高いもの」を把握したい人

### 補足

## 6. 週の定義

- 週は **カレンダー週** とする
- **月曜始まり** とする
- 記事の所属週は **公開日 (`published_at`) ベース** で判定する
- `published_at` が取得できない場合は `fetched_at` をフォールバックとして利用してよい

### 表示形式
週タイトルは以下の 2 段構成とする。

- `2026年3月第4週`
- `2026/03/23 - 2026/03/29`

## 7. 画面仕様

### 7.1 フィード管理画面

登録済みフィードの一覧表示・追加・削除を行う画面を用意する。

#### 機能
- フィード一覧の表示（name, url, type, is_active）
- フィードの新規追加
- フィードの削除

### 7.2 週次ダイジェスト画面

#### 構成
1. 週タイトル
2. 週全体の短いサマリー
3. 注目トピック
4. 記事一覧

#### 週全体の短いサマリー
- 長さは 2〜4 文
- 含める内容
  - 今週多かった話題
  - 特に読む価値が高い記事の傾向

#### 注目トピック
- 3〜5 個表示する
- タグ形式で表示する
- 件数表示は任意

#### 週の切り替え
- MVP では **最新週のみ** 表示する
- 前週・翌週へのナビゲーションは将来対応とする

#### 記事一覧の並び順
- **総合スコア順** とする

---

### 7.3 記事カード

各記事カードには以下を表示する。

- タイトル
- 出典名
- 公開日
- 短い要約 (`short summary`)
- 総合スコア
- 理由タグ 2〜3 個
- OG Image が取得できればサムネイル表示
- OG Image がなければプレースホルダー表示

#### リンクの挙動
- 記事タイトルをクリックすると元記事を **別タブで開く**
- アプリ内に記事詳細画面は設けない

#### サムネイルの扱い
- OG Image は補助的な視覚情報として扱う
- 取得できない場合でも記事カード自体は成立すること
- MVP における優先度は低め

- Web 標準仕様や新しい JavaScript / CSS 仕様にも関心がある
- AI 関連では、流行紹介よりも UI 実装・検証ワークフローにどう組み込むかを扱う記事を優先する

### 8.3 関心トピック

## 9. 注目トピックの仕様

### 9.1 方針
注目トピックは **分類ラベル型 (パターンA)** を採用する。

### 9.2 ルール
- 各記事に 1〜3 個のトピックラベルを付与する
- ラベルは **事前定義した候補集合** から選ぶ
- 週次ダイジェストでは、**その週に多く出現し、かつ高スコア記事にも含まれるトピック** を 3〜5 個選んで表示する

### 9.3 表示形式
- タグ形式
- 件数表示は任意

### 9.4 初期候補集合

## 10. スコアリング仕様

総合スコアは **層1 + 層2** の合成で決定する。

### 10.1 層1: 記事自体の価値
記事そのものの技術的価値を評価する。

#### 評価軸
- 具体性
- 深さ
- 実用性
- 新しさ

### 10.2 層2: ユーザー希望との一致
記事がユーザープロファイルにどれだけ合っているかを評価する。

#### 評価軸
- トピック一致
- 記事タイプ一致
- 深さ一致
- 実務関連性

### 10.3 スコアの値域
- 総合スコア、層1スコア、層2スコアはすべて **0〜100** の整数とする

### 10.4 推奨重み
- 層1: 45%
- 層2: 55%

### 10.5 減点対象
- 採用広報
- 入門まとめ

### 10.6 表示用情報
各記事について、以下を保持する。

- 総合スコア
- 層1スコア
- 層2スコア
- 理由タグ 2〜3 個
- 短い説明 1 文（任意）

### 10.7 一覧での見せ方
一覧には以下を表示する。

- 総合スコア
- 理由タグ 2〜3 個

スコア詳細の内訳数値を UI で大きく見せる必要はない。

## 11. 処理パイプライン

1 本の記事が週次ダイジェスト上の 1 カードになるまでの流れを以下とする。

1. ソースから記事候補を収集（全件）
2. URL ベースで重複排除
3. 本文抽出（**対象週の記事のみ**）
4. 記事単位で LLM 処理（**対象週の記事のみ**）
5. 週単位で LLM 処理
6. 週次ダイジェスト画面に表示

### 週スコープの方針
- 収集（ステップ 1〜2）は全件を対象とする。RSS フィードは時間経過で古い記事が消えるため、取りこぼし防止のために常に全件収集する。
- 本文抽出・LLM 処理（ステップ 3〜4）はコストが高い（HTTP fetch、LLM API 呼び出し）ため、対象週の記事のみに限定する。
- 記事の週への所属判定は `published_at` を基準とし、取得できない場合は `fetched_at` をフォールバックとする（セクション 6 と同一ルール）。

---

## 12. 記事単位の処理

### 12.1 収集
収集時に取得する最小情報:

- title
- url
- published_at
- source / feed
- author（取得できれば）

### 12.2 重複判定
- 同一 URL は同一記事とみなす
- URL 正規化は軽く行ってよい
  - trailing slash の吸収
  - 不要な query parameter の除去

### 12.3 本文抽出
記事 URL から本文情報を抽出する。

保持対象:
- cleaned content text
- cleaned content html
- title（再取得してもよい）
- og image
- metadata

### 12.4 記事単位の LLM 処理
各記事に対して独立に LLM を呼び出し、以下を生成する。

- short summary
- medium summary
- topics
- 層1スコア関連情報
- 層2スコア関連情報
- 総合スコア
- reason tags
- short reason（任意）

#### LLM 呼び出し方針
- 上記すべてを **1 回の LLM 呼び出しでまとめて生成** する

### 12.5 並列実行
記事単位の LLM 処理は、記事ごとに独立しているため **並列実行可能** とする。

---

## 13. 週単位の処理

### 13.1 週単位の LLM 処理
対象週に属する処理済み記事を入力として、週に対して 1 回だけ LLM を呼び出し、以下を生成する。

- 週全体の短いサマリー
- 注目トピック

### 13.2 キャッシュとしての扱い
- 週サマリーと注目トピックは **週次ダイジェスト表示を補助する派生データ / キャッシュ** とする
- 独立した週報成果物としては扱わない
- 対象週の記事構成が変化した場合に再生成可能とする

---

## 14. 失敗時の基本方針

MVP では複雑なフォールバックは行わない。

### 方針
- 本文抽出、記事単位の LLM 処理、週単位の LLM 処理は独立した処理段階として扱う
- 各処理段階でエラーが発生した場合は、ログおよび永続ストレージに記録する
- 記事単位の LLM 処理に失敗した記事は、週次ダイジェストの表示対象に含めない
- スコア生成に失敗した場合もフォールバック表示は行わない
- 失敗した処理は後からリトライ可能であること
- 週サマリー生成に失敗した場合でも、記事一覧自体は表示できるようにする

### 補足
エラーハンドリングの詳細設計は MVP 実装後に必要に応じて見直す前提とする。

## 17. 実装方針

### 17.1 アプリ構成
- Next.js をフロントエンド兼バックエンドとして使用する
- データアクセスは Prisma を利用する
- 永続化先は PostgreSQL とする
- 記事処理パイプラインは Server Actions またはサーバー側のジョブ実行基盤で扱う

### 17.2 パイプラインの起動方法
パイプライン（収集 → 本文抽出 → LLM 処理 → 週サマリー生成）は以下の 2 通りで起動できる。

- **npm script による CLI トリガー** — 引数なしの場合は今週、`2026-W13` のように weekKey を指定した場合は当該週を対象とする
- **Web UI 上のボタンによるトリガー** — 表示中の週を対象として動作する

いずれの場合も、収集は全件、本文抽出・LLM 処理・ステータス表示・リトライは対象週に限定される（セクション 11 参照）。

### 17.3 本文抽出
- 本文抽出には Readability 系ライブラリを使用する（ライブラリ選定は実装時に判断）

### 17.4 PostgreSQL
- ローカル開発環境では Docker で PostgreSQL を起動する前提とする

### 17.5 LLM の責務
LLM は以下に限定して使用する。

- 記事要約
- 記事トピック付与
- 記事スコアリング補助
- スコア理由生成
- 週サマリー生成
- 注目トピック生成

### 17.6 非 LLM 領域
以下は通常のアプリケーションロジックで扱う。

- RSS 収集
- URL 重複判定
- 週の計算
- 記事の週への割り当て
- 総合スコア順ソート
- 画面表示
- プレースホルダー制御
